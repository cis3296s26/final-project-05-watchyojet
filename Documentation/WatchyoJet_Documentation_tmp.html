1. System Overview

WatchyoJet is an autonomous Air Traffic Control (ATC) system that simulates aircraft movement, detects spatial conflicts between aircraft, and resolves them using deterministic, priority-based decision logic — all without human intervention.

The system is engineered around a clean, unidirectional processing pipeline:

┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   [ Simulation ]  →  [ Detection ]  →  [ Resolution ]  →  [ Update ] │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
Stage Responsibility
Simulation Maintain aircraft state; advance positions over time
Detection Identify pairwise conflicts using spatial proximity rules
Resolution Apply rule-based logic to resolve identified conflicts
Update Commit resolved states back to the simulation

WatchyoJet is designed to evolve into a real-time, production-grade ATC assistance platform — extensible toward live data feeds, trajectory prediction, and full-stack web visualization.

2. Currently Implemented System

2.1 Core Components

🛩️ Aircraft — Domain Model

The central entity representing a single aircraft in the simulation.

Field Type Description
callsign String Unique identifier (e.g., "AA101")
lat double Current latitude (decimal degrees)
lon double Current longitude (decimal degrees)
altitude double Current altitude in feet
speed double Speed in knots
heading double Direction of travel (0–360°)
type AircraftType Performance envelope (max/min altitude)
category AircraftCategory Priority classification

AircraftType — Defines performance boundaries per aircraft class:

  • Maximum cruising altitude
  • Minimum operating altitude
  • Performance-based constraints used during resolution

AircraftCategory — Priority enumeration used to determine which aircraft yields in a conflict:

EMERGENCY  >  MILITARY  >  CIVILIAN

🗂️ AircraftManager — State Container

Central registry that holds the live collection of all active Aircraft objects. Acts as the single source of truth for the simulation state. All engines read from and write to this manager.

🔄 MovementEngine — Position Propagator

Advances each aircraft's position forward in time by computing displacement from its current heading and speed.

Algorithm (per tick):

Δlat = speed × cos(heading) × Δt
Δlon = speed × sin(heading) × Δt
aircraft.lat += Δlat
aircraft.lon += Δlon

This is a simplified flat-earth approximation suitable for simulation-scale distances.

⚠️ ConflictDetector — Spatial Conflict Analyzer

Performs pairwise comparison of all aircraft to identify conflicts. Returns a List<Conflict> representing all active violations.

Detection Criteria (current):

  • Two aircraft are in conflict when their 3D proximity falls below the defined separation threshold.
  • Comparisons are exhaustive: every aircraft pair (i, j) where i < j is evaluated.

📋 Conflict — Conflict Model

Represents a detected conflict between exactly two aircraft.

Field Type Description
aircraft1 Aircraft First aircraft in the conflict pair
aircraft2 Aircraft Second aircraft in the conflict pair
severity double Calculated proximity / urgency score

⚙️ ResolutionEngine — Rule-Based Decision Maker

The core decision logic unit. For each Conflict, it:

  1. Selects the target aircraft — the lower-priority aircraft yields:

    EMERGENCY > MILITARY > CIVILIAN

    In case of equal priority, the engine applies a deterministic tiebreaker.

  2. Generates altitude options — four candidates evaluated in order:

    +1000 ft  |  -1000 ft  |  +2000 ft  |  -2000 ft

  3. Validates each candidate against hard constraints:

    Constraint Rule
    Maximum altitude newAlt ≤ aircraft.type.maxAltitude
    Non-negative altitude newAlt > 0
    Vertical separation buffer |newAlt - otherAlt| ≥ SAFE_VERTICAL_SEPARATION

  4. Applies the first valid resolution and outputs a Resolution object.

📄 Resolution — Resolution Model

Encapsulates a single resolution action taken by the engine.

Field Type Description
aircraft Aircraft The aircraft receiving the instruction
newAltitude double Assigned new altitude
reason String Human-readable explanation

🧠 ATCEngine — Decision Orchestrator

The top-level coordinator that drives the full detect → resolve → update cycle each tick.

ATCEngine.tick()
│
├── ConflictDetector.detect()    → List<Conflict>
│
├── ResolutionEngine.resolve()   → List<Resolution>
│
└── AircraftManager.applyAll()   → Updated State

3. Recent Improvements

3.1 Vertical Separation Enforcement

Change: The ResolutionEngine now validates that the proposed new altitude achieves meaningful vertical separation before applying a resolution.

if |proposedAltitude - otherAircraftAltitude| >= SAFE_VERTICAL_SEPARATION:
    → apply resolution
else:
    → skip (already safe or insufficient separation)

Why this matters:

Problem (Before) Solution (After)
Resolutions applied redundantly Skip resolutions when separation is already met
Aircraft climbed infinitely Hard ceiling via max altitude constraint
Control loop never converged Termination condition guaranteed each cycle
Unstable simulation behavior Predictable, stable state transitions

This single change was foundational: it transformed the resolution loop from a potentially divergent system into a convergent control loop.

3.2 Improved Test Coverage

MovementEngine Tests:

  • Replaced brittle exact-equality assertions with displacement-based validation
  • Tests now verify that aircraft moved in the correct direction by at least a minimum threshold distance
  • Resilient to floating-point precision drift

ConflictDetector Tests:

  • Added unit tests for aircraft pairs that:
    • Are clearly in conflict (within separation minimum)
    • Are clearly safe (far apart)
    • Are on the boundary of the separation threshold
  • Tests validate both the number and identity of detected conflicts

4. Current Limitations

WatchyoJet is a deliberate, iterative build. The following limitations reflect the current scope, not the target state:

Limitation Description
Reactive, not predictive Conflicts are resolved after they are detected, not before they occur
Vertical separation only No horizontal separation rules are enforced (e.g., 5 nautical miles)
No trajectory prediction The system cannot anticipate future positions or pre-emptively intervene
No real-time input Aircraft state is internal-only; no live data feeds or external sources
No frontend No map visualization or user interface exists yet
No persistence layer No database; all state is in-memory and lost between runs

6. Future Features

6.1 TrajectoryPredictor

Transform the system from reactive to proactive.

  • Compute predicted positions for each aircraft N seconds into the future
  • Run conflict detection on the predicted trajectory, not current position
  • Issue preemptive resolution instructions before aircraft enter conflict zones
  • Enables smooth, anticipatory control rather than emergency corrections

6.2 Horizontal Separation

Implement the real-world ICAO standard of 5 nautical miles lateral separation.

  • Extend ConflictDetector to compute 2D horizontal distance using the Haversine formula
  • A conflict is raised when: horizontalDist < 5nm AND verticalDist < 1000ft
  • Resolution engine generates heading-based and speed-based alternatives
  • This closes the largest gap between simulation and real ATC practice

  • Enables the frontend map to update in real time without polling

6.5 Real-Time Simulation Control

  • Inject new aircraft mid-simulation via API or UI
  • Remove aircraft (landed, diverted, etc.)
  • Modify flight parameters (altitude, heading, speed) dynamically
  • Pause / resume / reset the simulation clock
  • Adjustable tick rate for fast-forward or real-time modes

6.6 Airports & Airspace Zones

  • Define Airport entities with ICAO codes, runway layout, and approach corridors
  • Create Areas of Responsibility (AoR) — bounded airspace sectors
  • Assign aircraft to sectors; handoff logic between sectors
  • Define approach and departure sequencing per airport
  • Model restricted zones (military, no-fly) as exclusion polygons

Move beyond altitude-only resolution:

8. Engineering Quality & Standards

Design Principles

Principle Application
Single Responsibility Each engine owns exactly one concern (detect / resolve / move)
Dependency Injection ATCEngine receives its collaborators; testable in isolation
Immutable Models Conflict and Resolution are value objects; no side effects
Fail-Safe Defaults No resolution is applied if no valid option exists
Convergent Control Loop Vertical separation check ensures the loop always terminates

Test Strategy

Component Test Type Approach
MovementEngine Unit Displacement-based assertions (float-safe)
ConflictDetector Unit Boundary value testing (within / at / beyond sep)
ResolutionEngine Unit Priority logic + constraint validation
ATCEngine Integration Full cycle test with known initial state

Code Quality Targets

  • All public methods documented with Javadoc
  • No magic numbers — all constants named and centralized (e.g., SAFE_VERTICAL_SEPARATION)
  • Separation of concerns enforced at the package level
  • All resolution decisions logged with structured reasoning strings
