Section 1 — Introduction

What is WatchyoJet?

An autonomous ATC decision system that monitors real-world flight data, detects aircraft on converging paths, and automatically issues altitude separation instructions with no human input required.

Section 1 — The Problem

What problem does it solve?

Today's ATC

  • 100% reliant on human controllers
  • Controllers manage 10–20 aircraft at once, under constant pressure
  • Human fatigue leads to real incidents like runway incursions and near-misses
  • Separation decisions happen reactively, not predictively
  • No system-wide awareness of all converging pairs simultaneously

WatchyoJet

  • Tracks every aircraft in the airspace simultaneously
  • Uses CPA math to detect conflicts 2 minutes ahead
  • Resolves every conflict in under 2 seconds
  • Never fatigued, never distracted
  • Every decision is logged with callsigns and altitudes
Section 1 — Competitor Analysis

Existing Products

Leidos STARS

The FAA's actual Standard Terminal Automation Replacement System. Full radar integration, certified for operational use. Costs tens of millions to deploy.

Eurocontrol SESAR

European ATC modernisation initiative. Consortium of 60+ organisations. Focuses on pan-European airspace management at a continental scale.

FlightAware / Flightradar24

Flight tracking and situational awareness tools. Show where planes are but make no decisions — purely informational.

Saab Sensis ATC Systems

Enterprise ATC surveillance and automation. Used at major international airports. Proprietary, extremely expensive, requires hardware integration.

Section 1 — Differentiation

How WatchyoJet is different

  • ✦  Shadow mode: runs alongside real operations with no radio access needed
  • ✦  Open data: uses the free OpenSky Network API with zero licensing cost
  • ✦  Predictive: catches conflicts before they happen using CPA trajectory math
  • ✦  Transparent: every decision is logged with callsigns, altitudes, and timestamps
  • ✦  Category-aware: emergency aircraft are never moved and military gets priority

STARS / SESAR

Certified operational systems requiring radar hardware, millions in infrastructure, and FAA/EASA certification

WatchyoJet

Runs on a laptop. Uses public ADS-B data. Shows what autonomous ATC can look like and actually makes decisions in real time.

Section 5 — Challenges & Solutions

Technical Challenges

Resolutions were being overwritten by the live API

Every 12s refresh replaced the full aircraft state from raw transponder data, silently undoing every altitude change the engine issued.

Fix: AircraftManager now only updates lat/lon/speed/heading. Altitude is preserved once an aircraft is being tracked.

CRUISE-phase aircraft were all UNRESOLVED (24/24 conflicts)

The CRUISE search floor was set to FL180, but most cruise aircraft were between 10k–18k ft. This made searchFloor > searchCeiling — an empty search range. Every resolver call returned null.

Fix: CRUISE defaults to MIN_ALTITUDE (3,000 ft). Band lock then raises the floor only for aircraft already above FL180.

Dense airspace blocked every candidate altitude (9–16 UNRESOLVED)

With PROXIMITY_NM = 15, the isAltitudeFree check covered 15–20 aircraft around each target. Each blocked a 1,000 ft slot, filling the entire searchable band.

Fix: Reduced to 8 NM which is realistic for actual separation checks. Dropped blocked slots to 3–6 per aircraft.

Section 5 — Challenges & Solutions

More Challenges

Two aircraft assigned the same altitude in one cycle

Without coordination, two separate conflict pairs could both resolve to FL210, creating a new collision in the process of fixing the original ones.

Fix: pendingAltitudes map tracks every committed altitude within a cycle. isAltitudeFree() checks it before approving a candidate.

JavaFX WebView JS bridge called before the page loaded

The first implementation started the ATC loop immediately on launch. executeScript() calls were silently dropped. The map appeared but never updated.

Fix: Main loop now starts only after the WebView fires Worker.State.SUCCEEDED, confirming JS functions exist.

Non-technical: Scope creep and feature prioritisation

The original plan included weather integration, runway scheduling, heading-change advisories, and multi-airport support. All underestimated.

Fix: Scoped to what could be proven correct — vertical separation over one airspace. Everything else became future work.

Section 6 — Future Plans

If we had more time

Horizontal Resolution

Instead of only changing altitude, issue heading changes to move aircraft laterally away from each other. In real ATC, altitude changes are actually a last resort.

Weather Layer

Pull METAR/TAF data and factor wind, turbulence, and precipitation into routing. Currently all aircraft are treated as flying in calm conditions.

Runway Sequencing

Build a landing queue for PHL — sequence aircraft by arrival time, manage approach spacing, and assign runways automatically based on wind direction.

Voice Output

Generate synthesised ATC radio calls like "United 347, climb and maintain flight level two-four-zero" to simulate what a real controller would say.

Section 7 — Personal Reflections

What we each took away

The hardest part was debugging silent failures in the JS bridge. No error, no output, just nothing happening. Taught me to always verify the bridge is ready before calling into it. Mateusz

The map was the most satisfying part to build. Watching plane icons actually move on a real map felt great. Getting the movement to interpolate smoothly between API updates was harder than I expected though. Next time I'd start on the trajectory math earlier instead of pushing it to later in the sprint.

Dylan

Managing the board taught me how fast scope grows when everyone has ideas. Keeping sprints realistic was harder than I thought it would be. Working in a team takes more coordination than going solo but we built something that none of us would have finished on our own in this timeframe.

Section 8 — Project Management

Scrum & Teamwork

How we ran the project

  • Weekly Scrum meetings — every Friday, in person
  • Features developed throughout the week, discussed at each meeting
  • Project board and sprint report updated every week
  • Quality check at each meeting: demo new functionality, decide if branch is ready to merge to main
  • Feature branches — PRs reviewed before any merge

Reflection on teamwork

  • Having a fixed weekly meeting kept everyone accountable
  • The merge gate at meetings caught problems before they hit main
  • Scope was the hardest thing to agree on — everyone had more ideas than time
  • Working as a team meant we shipped something more complete than any one of us could have alone

The biggest non-technical lesson: scoping a project is harder than building it. We planned features for a production ATC system — and built a working prototype that proves the concept.
